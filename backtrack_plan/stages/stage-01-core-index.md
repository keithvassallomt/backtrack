# Stage 1 — Core Index (the catalogue)

## Objective
`backtrack-core` can build and query the SQLite catalogue that the entire UI runs on.
After this stage you can ingest borg archive listings and answer, instantly and
offline: "what was in this folder at snapshot N?", "what's this file's history?",
"which files changed since archive X?", and filename search including deleted files.

## Prerequisites
Stage 0. Read [../reference/stack.md](../reference/stack.md) §3 (schema) and
[../reference/open-questions.md](../reference/open-questions.md) Q3 (why the index
exists — the performance cliffs of talking to Borg directly).

## Key design constraints (do not violate)
- **Interval encoding**: one `versions` row per file-version spanning
  `first_seq..=last_seq`, NOT one row per file×archive. 500k files × 100 archives
  must stay ≈ 1M rows / ≲150 MB.
- **Single writer**: all writes go through one `IndexWriter` type (the daemon will
  own it). Readers open the DB read-only. WAL mode, `synchronous=NORMAL`.
- Diffing between consecutive archives happens **in SQL against the previous
  archive's rows**, never via `borg diff`.

## Tasks

### S01-T1 — Schema + migrations
Implement the schema from stack.md §3 (archives, paths, versions, fts_names, meta)
with a `schema_version` in `meta` and forward-only migration runner. On open:
`PRAGMA integrity_check` (quick) — failure surfaces as typed `IndexCorrupt` error.
Indexes: `versions(path_id, first_seq, last_seq)`, `paths(parent_id, name)` unique.
**Accept:** unit tests create/migrate/reopen; corrupting the file yields `IndexCorrupt`.

### S01-T2 — Ingest
`IndexWriter::ingest_archive(archive_meta, impl Iterator<Item=BorgItem>)` where
`BorgItem` is the parsed `borg list --json-lines` record (path, type, size, mtime,
mode, and the chunk-derived content hash when available). Streaming: constant
memory, one transaction, ≥20k items/sec on dev hardware. Logic per item:
resolve/insert path chain; compare against the previous archive's version row
(size+mtime+hash); unchanged → extend `last_seq`; changed/new → close old interval,
open new row. Items present before but absent now → close their intervals.
Fixtures: generate JSONL fixture files (small: 100 files; large: 200k files) checked
into `testdata/` (large one generated by a script, gitignored, built by `just demo-repo`).
**Accept:** property test — ingesting the same listing twice changes nothing;
synthetic churn sequences produce exactly the expected row counts; large-fixture
ingest under 15s in CI.

### S01-T3 — Timeline queries
Read-side `IndexReader` with:
- `folder_at(path, seq) -> Vec<Entry>` (the main file-pane query; includes per-entry
  status flags: `deleted_after(seq)`, `changed_since(seq)` vs latest),
- `file_history(path) -> Vec<VersionSpan>`,
- `archives_overview()` (grouped list data: counts per day/week/month, spool/local
  flags),
- `next_change(path, from_seq, direction)` (powers "next change to selected file").
**Accept:** each query has unit tests against a scripted history fixture with known
answers; `folder_at` on the large fixture < 10 ms.

### S01-T4 — Search
FTS5 over file names (insert on first appearance; keep rowid ↔ path_id mapping).
`search(query) -> Vec<SearchHit>` where each hit aggregates per-file: full path,
lifespan (first/last seq + timestamps), version count, exists-today flag, kind.
Ranking: exists-today=false first, then recency of last existence, then bm25.
**Accept:** Charlie test — file created in fixture archives 10–40, absent since:
search finds it, flags it deleted, reports the correct lifespan; prefix and
substring-ish queries (`contract`, `cont*`) behave.

### S01-T5 — Changed-since query
`changed_since(seq) -> Vec<PathBuf>`: files whose current on-disk state (given a
walker that stats the live filesystem against the archive-`seq` rows) differ by
size/mtime. This feeds the offline spool (Stage 5). Keep the filesystem walk
separate from the SQL so it's testable with a fake walker.
**Accept:** fixture test with a temp dir: touch/modify/add/delete files → exactly
the right set returned.

### S01-T6 — Prune/expiry
`remove_archives(seqs)`: close/merge intervals as if those archives never existed
(a version spanning a removed seq must not split wrongly; spans merge when their
surrounding archives collapse together). Used by prune (Stage 4) and spool expiry
(Stage 5).
**Accept:** property test — ingest N archives, remove a random subset, resulting
queries equal a from-scratch ingest of the surviving archives.

### S01-T7 — `just demo-repo`
Script (rust bin or shell in `xtask/`) that builds `~/.local/share/backtrack-dev/`:
creates a real borg repo, seeds a scripted 30-snapshot history over a fake home
(files appearing/changing/disappearing on known dates — reuse the names from the
mockups: `report.odt`, `invoice-may.pdf`, `old-client-folder`, …), ingests it into
a fresh index. This is the dataset every GUI stage develops against.
**Accept:** `just demo-repo` completes < 2 min; the Stage 6 timeline shows
`old-client-folder` with a "deleted after this" badge at the right snapshots.

## Definition of Done
All tests green in CI; ingest/query perf numbers recorded in the stage's progress
notes; progress.md updated; CHANGELOG untouched (no user-visible change yet — this
is allowed for pure-internal stages, note it in the commit body instead).
